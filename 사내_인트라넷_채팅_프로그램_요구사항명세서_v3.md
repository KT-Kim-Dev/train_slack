# 사내 인트라넷 채팅 프로그램 요구사항 명세서 (PoC) — v3.0

- **문서 버전**: v3.0
- **작성일**: 2026-07-03
- **작성 목적**: Cursor AI를 활용한 PoC 구현을 위한 개발 요구사항 정의
- **프로젝트명(가칭)**: Intra-Chat
- **v3.0 변경사항**: AI 채팅(로컬 Ollama 연동), Yona 이슈 관리 연동, Jenkins 빌드 연동 기능 추가. 클라이언트/서버 기술 스택은 기존 결정대로 **Electron + Node.js** 유지 (C#/.NET 미채택)

---

## 1. 프로젝트 개요

### 1.1 배경
현재 사내는 인트라넷(외부망 차단) 환경으로, 카카오톡 등 외부 개인 메신저를 업무에 사용할 수 없다. 메일 외에는 실시간 소통 및 대화 기록 보관 수단이 없어, 이를 대체할 사내 전용 채팅 프로그램이 필요하다. 나아가 단순 채팅을 넘어, 사내에서 사용 중인 **Yona(이슈 관리 툴)** 및 **Jenkins(빌드/CI)** 와 연동하고, 로컬 LLM(Ollama) 기반 **AI 채팅**까지 하나의 프로그램에서 제공하고자 한다.

### 1.2 목적
슬랙(Slack)의 핵심 기능(채널, 그룹채팅, 1:1 DM, 실시간 메시징, 파일 공유)에 더해, 개발/운영 업무에 필요한 **이슈 조회·생성(Yona), 빌드 실행·조회(Jenkins), 사내 AI 어시스턴트(Ollama)** 기능을 통합한 사내 업무 커뮤니케이션 플랫폼을 인트라넷 환경에서 동작하는 설치형 Windows 데스크톱 프로그램으로 구현한다. 실제 슬랙 데스크톱 앱과 동일하게 **Electron 기반**으로 구현한다.

> 참고: 검토 과정에서 C#/.NET 기반 아키텍처안도 제안되었으나, 팀 내 Node.js/Electron 개발 경험 및 기존 결정을 존중해 **Electron + Node.js 스택을 그대로 유지**하기로 확정한다. Yona/Jenkins/Ollama와의 연동은 모두 Node.js 서버에서 REST API 호출 방식으로 처리한다.

### 1.3 목표 사용자 규모
- 최소 10명, 최대 20명 동시 사용
- 전 사용자 Windows PC 사용, 인트라넷 환경에 위치

### 1.4 PoC 성공 기준
- 10~20명이 동시 접속한 상태에서 메시지 및 파일 전송이 지연 없이 이루어질 것
- Yona 이슈 조회/생성, Jenkins 빌드 실행/조회, AI 채팅이 채팅 인터페이스 내에서 명령어를 통해 정상 동작할 것

> ⚠️ **일정 관련 유의사항**: 기존에는 "채팅 핵심 기능만" 기준으로 1~2주 PoC 일정을 잡았다. 이번 버전에서 Yona/Jenkins/AI 연동이 추가되면서 범위가 크게 늘었으므로, 아래와 같이 **단계적(Phase) 진행**을 권장한다.
> - **Phase 1 (1~2주)**: 채널/그룹/DM/파일 송수신 등 채팅 핵심 기능 (기존 범위)
> - **Phase 2 (추가 1~2주 이상)**: AI 채팅, Yona 연동, Jenkins 연동
>
> 전체를 한 번에 1~2주 안에 끝내는 것은 현실적으로 어려우므로, 관리자와 일정 재조율이 필요하다.

## 2. 범위 (Scope)

### 2.1 포함 범위 (In Scope)

**[Phase 1] 채팅 핵심 기능**
- 관리자 계정 발급 기반 로그인
- 채널(공개), 그룹채팅(비공개), 1:1 DM
- 실시간 텍스트 메시지 송수신
- 메시지 히스토리 조회(스크롤 페이징)
- 파일/이미지 첨부 및 송수신 (업로드/다운로드/미리보기)
- 접속 상태(온라인/오프라인) 표시

**[Phase 2] 업무 연동 기능 (신규)**
- AI 채팅 (로컬 Ollama 연동, 명령어 또는 멘션 기반)
- Yona 이슈 조회 / 생성
- Jenkins 빌드 실행 / 조회 / 완료 알림

### 2.2 제외 범위 (Out of Scope, PoC 단계)
- 회원가입(셀프 가입) 기능
- 음성/영상통화, 화면 공유
- 스레드 답글, 세분화된 이모지 리액션
- 고도화된 검색(형태소 분석 등)
- 코드 리뷰 자동화, 로그 자동 분석, RAG(사내 문서 기반 검색) 등 고급 AI 기능 — **PoC에서는 제외, 향후 확장 과제로 분류**
- Jenkins 배포 승인 워크플로우(관리자 승인 절차) — **PoC에서는 조회/실행까지만, 승인 플로우는 향후 확장 과제로 분류**
- 클라이언트 자동 업데이트 기능
- SSO/AD·LDAP 연동 인증
- Slack 앱/Slash Command 자체(외부 Slack 사용 안 함 — 자체 클라이언트 내 명령어로 대체)

| 항목 | 내용 |
|---|---|
| 관리자 | 경태 (계정 발급/삭제 권한 보유) |

| 보안/기록물 관리 규정 | 별도 사내 규정 없음 (자유 설계 가능하나 최소한의 로그는 권장) |
| 연동 대상 시스템 | Yona(이슈 관리), Jenkins(빌드/CI), Ollama(로컬 LLM 서버) — **모두 인트라넷 내에서 접근 가능해야 하며, 각 시스템의 API 접근 정보(URL, 인증 토큰 등) 사전 확인 필요** |

## 4. 시스템 아키텍처

```
[내부망 서버 1대 - Windows, 다른 업무와 공유 사용 중]

  ┌───────────────────────────────────────────┐
  │            Node.js + Express               │
  │        (REST API Gateway / 인증)            │
  └───────────────────┬────────────────────────┘
                       │
        ┌──────────────┼───────────────┬───────────────┐
        │              │               │               │
   Chat Service   Yona Service   Jenkins Service   AI Service
   (채널/DM/파일)   (이슈 조회/생성)  (빌드 실행/조회)   (Ollama 연동)
        │              │               │               │
   Socket.IO       Yona REST API   Jenkins REST API   Ollama API
   (실시간 메시지)                                    (OpenAI 호환, 스트리밍)
        │
     SQLite (사용자/채널/메시지/파일/명령로그/빌드이력)

[클라이언트 - Windows PC 10~20대]
  └─ Electron + React 기반 데스크톱 앱 (exe)
      ├─ 로그인 화면
      ├─ 사이드바: 채널 / 그룹채팅 / DM / AI 채팅 목록
      ├─ 채팅창: 실시간 메시지 (Socket.IO 수신)
      ├─ 파일 업로드/다운로드 UI
      ├─ 명령어 입력 처리 (/issue, /build, /ai 등)
      └─ 카드형 UI: 이슈 정보, 빌드 상태 표시용 컴포넌트
```

- 서버는 Chat/Yona/Jenkins/AI 기능을 **서비스 레이어로 분리**하여, 이후 다른 사내 시스템(GitHub, SonarQube 등)도 쉽게 추가할 수 있도록 모듈화한다.
- Yona, Jenkins, Ollama 서버가 채팅 서버와 같은 인트라넷 대역에 있어 통신 가능한지 사전 확인이 필요하다 (본 서버는 이미 외부 인터넷은 불가하지만, 인트라넷 내 다른 서버와의 통신은 가능해야 함).

## 5. 기능 요구사항 (Functional Requirements) — 상세

### 5.1 인증/계정

| ID | 요구사항 | 상세 설명 | 우선순위 |
|---|---|---|---|
| FR-01 | 관리자 계정 생성 | 관리자는 서버 측 CLI 스크립트 또는 최소한의 관리자 화면을 통해 사용자 계정(아이디, 임시 비밀번호, 표시이름)을 생성할 수 있다 | 필수 |
| FR-02 | 로그인 | 사용자는 클라이언트 실행 후 로그인 화면에서 발급받은 ID/PW를 입력해 로그인한다. 로그인 성공 시 서버는 JWT 토큰을 발급한다 | 필수 |
| FR-03 | 비밀번호 보안 | 비밀번호는 bcrypt 등으로 해시하여 저장하며, 평문으로 저장/전송하지 않는다 | 필수 |
| FR-04 | 계정 비활성화/삭제 | 관리자는 특정 계정을 비활성화하거나 삭제할 수 있다 | 필수 |
| FR-05 | 자동 로그인(선택) | 클라이언트 재실행 시 저장된 토큰으로 자동 로그인을 시도한다 | 권장 |

|---|---|---|---|

| FR-10 | 채팅방 나가기 | 채널/그룹채팅에서 나갈 수 있다 | 권장 |

### 5.6 AI 채팅 (로컬 Ollama 연동) — 신규

| ID | 요구사항 | 상세 설명 | 우선순위 |
|---|---|---|---|
| FR-27 | AI 채팅방 | 사용자별로 AI와 1:1로 대화하는 전용 채팅방("AI 어시스턴트")을 자동 제공한다 | 필수 |
| FR-28 | 명령어/멘션 트리거 | 일반 채널/그룹에서도 `@ai` 멘션 또는 `/ai (질문)` 명령어로 AI를 호출할 수 있다 | 필수 |
| FR-29 | Ollama 연동 | 서버는 사용자의 질문을 내부망의 Ollama 서버(OpenAI 호환 API)로 전달하고 응답을 받아온다 | 필수 |
| FR-30 | 스트리밍 응답 | AI 응답은 완성 후 한번에 표시하지 않고, 생성되는 대로 채팅창에 순차적으로 업데이트되며 표시된다 | 권장 |
| FR-31 | 대화 컨텍스트 유지 | 같은 AI 채팅방(세션) 내에서는 이전 대화 맥락을 함께 전달하여 AI가 문맥을 이해하도록 한다 | 필수 |
| FR-32 | 대화 이력 저장 | AI와의 대화도 일반 메시지와 동일하게 DB에 저장되어 이후 스크롤로 조회 가능하다 | 필수 |
| FR-33 | 모델/옵션 선택(선택) | 관리자 또는 사용자가 사용할 로컬 모델을 선택할 수 있는 옵션을 둘 수 있다 (서버에 설치된 Ollama 모델 목록 중 선택) | 선택 |
| FR-34 | 예외 처리 | Ollama 서버 다운, 타임아웃, 모델 없음 등의 상황에서 사용자에게 명확한 오류 메시지를 표시한다 | 필수 |

### 5.7 Yona 이슈 관리 연동 — 신규

| ID | 요구사항 | 상세 설명 | 우선순위 |
|---|---|---|---|
| FR-35 | 이슈 조회 명령어 | `/issue (이슈번호)` 입력 시 Yona API를 통해 해당 이슈의 제목, 담당자, 우선순위, 상태, 마감일을 조회해 채팅창에 카드 형태로 표시한다 | 필수 |
| FR-36 | 이슈 생성 UI | `/issue create` 또는 별도 버튼 클릭 시 모달(제목, 설명, 담당자, 프로젝트, 라벨 입력 폼)이 열리고, 저장 시 Yona API로 이슈가 생성된다 | 필수 |
| FR-37 | 생성 결과 링크 반환 | 이슈 생성 완료 시 채팅창에 생성된 이슈로 바로 이동할 수 있는 링크가 표시된다 | 필수 |
| FR-38 | 인증/연동 정보 | Yona API 접근을 위한 인증 토큰/키는 서버 설정 파일에 안전하게 보관하며, 클라이언트에는 노출되지 않는다 | 필수 |
| FR-39 | 예외 처리 | 존재하지 않는 이슈번호 조회, Yona 서버 응답 없음 등의 상황에 대한 안내 메시지를 표시한다 | 필수 |

### 5.8 Jenkins 빌드 연동 — 신규

| ID | 요구사항 | 상세 설명 | 우선순위 |
|---|---|---|---|
| FR-40 | 빌드 실행 명령어 | `/build (프로젝트명)` 입력 시 확인창(실수 방지)을 거쳐 Jenkins API로 빌드를 실행 요청한다 | 필수 |

| 배포 | 클라이언트는 단일 실행파일(exe) 형태로 패키징, 공유폴더 업로드 방식으로 배포 |
| 유지보수성 | 서버/클라이언트 모두 JavaScript/TypeScript 기반으로 통일. Yona/Jenkins/AI 연동은 각각 독립된 서비스 모듈로 분리하여 향후 다른 시스템(GitHub 등) 추가가 쉽도록 한다 |

## 7. 데이터 모델 (요약)

```sql
-- 기존 채팅 관련 테이블
users(id, username, password_hash, display_name, is_online, last_seen, created_at)
rooms(id, name, type['channel'|'group'|'dm'|'ai'], created_by, created_at)
room_members(room_id, user_id, joined_at, last_read_message_id)
messages(
  id, room_id, sender_id,
  message_type['text'|'file'|'image'|'ai_response'|'card'],
  content, file_name, file_path, file_size,
  created_at
)

-- 신규: AI 채팅
ai_sessions(id, user_id, room_id, model_name, created_at)

-- 신규: 명령어 실행 로그 (Yona/Jenkins 공통)
command_logs(
  id, user_id, command, parameter,
  success, elapsed_ms, created_at
)

-- 신규: Jenkins 빌드 이력
build_history(
  id, project, build_number, status,
  duration_sec, started_at, finished_at, triggered_by
)
```

- `rooms.type`에 `'ai'`를 추가해 AI 전용 채팅방을 기존 구조에 자연스럽게 통합한다.
- Yona 이슈 데이터는 별도로 캐싱하지 않고 조회 시마다 Yona API를 직접 호출하는 것을 기본으로 한다 (PoC 단계에서는 데이터 정합성이 더 중요하므로).
- `command_logs`, `build_history`는 문제 추적 및 이력 확인용으로, 필수 로그 요구사항(FR 로그)을 뒷받침한다.

## 8. API 설계 (신규 연동 기능)

### 8.1 인증
```
POST /api/auth/login
Request  { "username": "...", "password": "..." }
Response { "token": "...", "user": { "id": 3, "displayName": "..." } }
```

### 8.2 AI 채팅
```
POST /api/ai/chat
Request  { "roomId": 12, "message": "이번 Sprint 목표 알려줘", "model": "gemma3" }
Response (스트리밍, Server-Sent Events 또는 WebSocket 이벤트로 순차 전송)
  { "delta": "현재 ", "done": false }
  { "delta": "Sprint 목표는...", "done": false }
  { "delta": "", "done": true, "elapsedMs": 2400 }
```

### 8.3 Yona 연동
```
GET  /api/yona/issues/:id
Response { "title": "...", "assignee": "...", "priority": "High", "status": "진행중", "dueDate": "2026-07-10" }

POST /api/yona/issues
Request  { "title": "...", "description": "...", "assignee": "...", "project": "...", "labels": [] }
Response { "issueId": 532, "url": "http://yona.internal/issues/532" }
```

### 8.4 Jenkins 연동
```
POST /api/jenkins/build/start
Request  { "project": "ProjectA" }
Response { "buildNumber": 135, "queuedAt": "2026-07-03T10:00:00" }

GET /api/jenkins/build/:project/status
Response { "status": "SUCCESS", "durationSec": 132, "logUrl": "http://jenkins.internal/job/ProjectA/135/console" }

POST /api/webhooks/jenkins   -- Jenkins → 서버로 빌드 완료 시 콜백
Request  { "project": "ProjectA", "buildNumber": 135, "status": "SUCCESS", "durationSec": 132 }
```

> Jenkins 완료 알림(FR-42)은 Jenkins가 우리 서버로 웹훅을 호출하는 방식을 기본으로 하며, Jenkins 쪽에서 웹훅 호출이 불가능한 환경이라면 서버가 주기적으로 상태를 폴링(polling)하는 방식으로 대체한다. **이 부분은 Jenkins 서버 설정 권한에 따라 결정되므로 사전 확인이 필요하다.**

## 10. 구현 기술 방식 추천 및 사유

| 영역 | 추천 기술 | 추천 이유 |
|---|---|---|
| 클라이언트 프레임워크 | **Electron + React** | 슬랙 데스크톱 앱과 동일한 베이스로, 채팅 UI뿐 아니라 이슈 카드/빌드 상태 등 다양한 리치 UI 컴포넌트를 구현하기에도 적합 |
| 서버 프레임워크 | **Node.js + Express** | 클라이언트와 동일한 언어로 개발 생산성 확보. 서비스 레이어(Chat/Yona/Jenkins/AI)를 라우터/모듈 단위로 깔끔하게 분리 가능 |
| 실시간 통신 | **Socket.IO** | 텍스트 메시지뿐 아니라 AI 스트리밍 응답, 빌드 완료 알림 등 서버發 실시간 푸시가 다양해진 만큼, room 기반 브로드캐스트가 강력한 Socket.IO가 적합 |
| 데이터베이스 | **SQLite (better-sqlite3)** | 20명 규모 + 명령어 로그/빌드 이력 정도의 데이터량은 SQLite로 충분히 처리 가능 |
| AI 연동 | **Ollama (OpenAI 호환 API)** | 로컬에서 구동되는 LLM이라 인트라넷 폐쇄망 환경과 완전히 호환됨. Node.js에서 `fetch`/`axios`로 OpenAI 호환 엔드포인트를 그대로 호출 가능해 별도 SDK 없이 연동 가능 |
| Yona/Jenkins 연동 | **Node.js axios 기반 REST 클라이언트** | 두 시스템 모두 REST API를 제공하므로 별도 SDK 없이 axios로 간단히 연동. 서비스 모듈로 분리해 필드 변경이나 API 버전 변경에도 유연하게 대응 |
| 인증 | **자체 발급 ID/PW + JWT** | 기존 결정 유지. 외부 시스템(Yona/Jenkins) 인증 정보는 사용자 토큰과 별개로 서버가 보관 |
| 클라이언트 패키징 | **electron-builder** | 기존 결정 유지 |

> **C#/.NET 미채택 사유**: ChatGPT 대화에서 제안된 .NET 8 + SlackNet 기반 아키텍처는 견고한 선택지이나, 팀의 기존 Electron/Node.js 개발 경험 및 이미 진행 중인 작업과의 일관성을 고려해 채택하지 않는다. 대신 동일한 서비스 분리(Chat/Yona/Jenkins/AI) 원칙은 Node.js 환경에서도 그대로 적용한다.

## 11. 개발 시작 전 설치/준비 목록

### 11.1 서버 측 준비

| 항목 | 설치/준비 방법 |
|---|---|
| Node.js (LTS 버전) | 인터넷 가능한 PC에서 설치파일(.msi) 다운로드 후 서버로 반입, 설치 |

| SQLite | 별도 설치 불필요 (better-sqlite3에 내장) |
| **Ollama** | 인터넷 가능한 PC에서 Ollama 설치파일 및 사용할 모델(GGUF)을 미리 `ollama pull`로 받아 모델 파일을 반입 후, 오프라인 서버에서 로컬 모델 경로로 등록 |

| 공유폴더 접근 권한 | 서버 프로세스의 읽기/쓰기 권한 사전 확인 필요 |

## 12. 리스크 및 고려사항

| 리스크 | 내용 | 대응 방안 |
|---|---|---|
| 일정 리스크 | Yona/Jenkins/AI 연동 추가로 기존 1~2주 목표를 넘어설 가능성이 높음 | Phase 1(채팅)과 Phase 2(연동)로 나누어 단계적으로 진행, 관리자와 일정 재조율 |

| 대용량 파일 전송 시 서버 부하 | 파일 업로드가 몰릴 경우 서버에 영향 | 파일 크기 제한(FR-20) 적용 |
